# 旅のしおり Web アプリ 仕様書

旅行のしおり（旅程表）を作成・共有できるシングルページWebアプリ。
ビルド不要・サーバー不要の静的ファイル構成。ブラウザで index.html を開くだけで動作する。

## ファイル構成

- `index.html` — マークアップ（UIの骨格、モーダル含む）
- `styles.css` — 全スタイル
- `i18n.js` — 4言語の翻訳辞書と言語切替ロジック（`window.I18N` を公開）
- `app.js` — アプリ本体（状態管理、描画、ドラッグ&ドロップ、ルート計算、共有、入出力、地図）
- `vendor/leaflet/` — Leaflet 1.9.4 を同梱（leaflet.js / leaflet.css / images/）。CDN参照は不可
- `vendor/firebase/` — Firebase JS SDK compat版 v12.16.0 を同梱（firebase-app-compat.js / firebase-auth-compat.js / firebase-firestore-compat.js）。Google ログイン＋Firestoreクラウド保存（15）用。CDN参照は不可

外部CDN・外部ライブラリのネットワーク読み込みは禁止（ライブラリは同梱する）。実行時の外部通信は Nominatim ジオコーディングAPIと OpenStreetMap タイル（`https://tile.openstreetmap.org/{z}/{x}/{y}.png`）のみ。
フレームワークなしのバニラJS（ES2020+、モジュール不使用で `<script src>` 読み込み）。

## デザイン方針

「旅のしおり」らしい、かわいくて見やすいデザイン:
- パステル調の配色、丸みのあるカード、手帳っぽい雰囲気
- 縦タイムライン形式: 左に時刻列、右にカテゴリー色のカードが並ぶ
- カテゴリーごとに色とアイコン(絵文字)を固定:
  - 移動 move: 青系 🚌 / 食事 meal: オレンジ系 🍽️ / 宿泊 stay: 紫系 🏨 / 観光 sight: 緑系 📸
- ヘッダーに旅行タイトル（クリックで編集可）と言語切替、日タブ
- スマホ幅(375px)〜PC幅までレスポンシブ対応
- ライト/ダーク両対応は不要。ライトのみで良い。

## データモデル

```js
trip = {
  v: 1,                      // スキーマバージョン
  title: "東京旅行",
  lang: "ja",                // ja | en | zh | th
  days: [
    {
      date: "2026-07-20",    // 省略可 ("")
      startTime: "09:00",    // その日の開始時刻
      tz: "",                 // IANAタイムゾーン文字列（例 "Asia/Tokyo"）。既定 ""＝時差計算なし（13参照）
      priv: false,           // その日を丸ごと非公開マーク（既定false。14参照）
      dateManual: false,     // true なら date をユーザーが手動で確定した日（自動連番の対象外。既定false。2参照）
      items: [
        {
          id: "ab12cd",              // ランダム短ID
          cat: "sight",              // move | meal | stay | sight
          name: "浅草寺",             // 項目名（滞在地名）
          loc: "",                   // 位置検索用の地名。空なら name を使う
          dur: 90,                   // 滞在(所要)時間・分
          note: "雷門で写真",         // メモ
          priv: false,               // 項目まるごと非公開マーク（既定false。move含む全カテゴリー共通。14参照）
          notePriv: false,           // メモのみ非公開マーク（既定false。move含む全カテゴリー共通。14参照）
          fixedStart: null,          // 開始時刻の手動固定（"HH:MM"文字列 or null。move含む全カテゴリー共通。17参照）
          lat: 35.714, lon: 139.796, // 座標（未取得なら null / 省略）
                                     // 共有リンクにも含まれ、受け取った側は再検索なしで地図表示できる
          names: {},                 // スポット名の多言語表示（3c）: { en?, zh?, th?, ja?: string|null }。
                                     // move含む全カテゴリー共通。move は「A → B」の組み立てができない
                                     // ときのフォールバック表示にのみ使う（3c参照）
          noteNames: {},             // メモの多言語表示（3c 追記）: names と同形。notePriv:true の項目は
                                     // 公開コピーで note と同様に空にする
          gmap: ""                   // move以外のみ: Google Maps 共有リンク（任意）。
                                     // 貼られたURLから座標を抽出でき、それが lat/lon の最優先ソースになる
          // 以下 move のみ:
          mode: "train",             // walk | train | bus | car | shinkansen | plane | ferry | other
          distKm: 12.3,              // ジオコーディングで得た直線距離(再計算用, null可)
          auto: true,                // ルート検討ボタンが生成した項目か
          arriveTz: ""               // 到着地のIANAタイムゾーン文字列（任意）。既定 ""＝時差なし（13参照）
        }
      ]
    }
  ],
  // 持ち物リスト・やることリスト（10）: しおり単位（日ごとではない）。
  // localStorage・共有リンクに含まれる（＝共有相手にも見える／持ち物の分担に使える）
  packing: [
    { id: "pk01ab", text: "パスポート", done: false, priv: false } // priv: 非公開マーク（既定false。14参照）
  ],
  todos: [
    { id: "td01ab", text: "モバイルWi-Fiを予約", done: true, priv: false }
  ],
  packingPriv: false, // 持ち物リスト全体を丸ごと非公開マーク（既定false。14参照）
  todosPriv: false    // やることリスト全体を丸ごと非公開マーク（既定false。14参照）
}
```

状態は変更のたびに `localStorage`（キー `tabi-shiori-v1`）へ自動保存。起動時に復元。

## 画面構成と機能

### 1. ヘッダー
- 旅行タイトル（インライン編集）
- 言語切替（🌐 セレクト: 日本語 / English / 中文 / ไทย）
- 「共有」「テキスト入出力」ボタン

### 2. 日タブ
- Day 1, Day 2, ... のタブ。「+ 日を追加」で追加、タブ長押し/×で削除（確認ダイアログ）
- 各日の日付と開始時刻を編集できる欄をタブ下に表示

#### 2a. 日付の自動連番

Day 1 に日付を入れると、Day 2 以降の日付が自動的に連番（前日+1日）で埋まる。もちろん個別に手動で変更することもできる。

- **データ**: `day.dateManual: boolean`（既定 false）を追加する。true なら、その日の `date` はユーザーが手動で確定した値であり自動連番の対象外になる。`normalizeTrip` で防御的に正規化する（boolean 以外の型は既定 false にフォールバック）
- **再計算ロジック（`recalcAutoDates`）**: index 1 以降の日を順に見て、`dateManual === false` かつ**直前の日（index-1）に有効な日付がある**なら、その日の `date` を「直前の日の日付 + 1日」で上書きする。直前の日の日付が空（または不正な形式）なら、その日の `date` も空のまま（連鎖を止める＝それ以降の自動の日も再計算のたびに空へ戻る）。**index 0（Day 1）は常にユーザー入力のアンカーとして扱い、このロジックの対象外**（`dateManual` の値に関わらず変更しない）
  - 「Day1の日付 + N日」ではなく「直前の日から常に+1日」方式にする。これにより、途中の日を手動で別の日付に飛ばしても、それ以降の自動の日は手動で飛ばした日を起点に自然に連番が続く
  - 日付の加算は必ずUTCベースで行う（ローカルタイムゾーンのDST切替等で日がずれるのを防ぐため）:
    ```js
    var m = /^(\d{4})-(\d{2})-(\d{2})$/.exec(dateStr);
    var d = new Date(Date.UTC(+m[1], +m[2]-1, +m[3]));
    d.setUTCDate(d.getUTCDate() + 1);
    return d.toISOString().slice(0, 10);
    ```
  - **呼び出しタイミング**: 日付入力欄の変更時（`change`）、「＋ 日を追加」の直後、日の削除の直後、の3箇所で呼び出す
- **手動フラグの立て方**（日付入力欄の `change` イベント時）:
  - index 0（Day 1）の編集では `dateManual` を一切変更しない（常にアンカーのまま）
  - index 1 以降の日の日付入力を編集して**空でない値**にしたら `dateManual = true` にする（自動計算の対象から外れる）
  - その入力を**空にしたら `dateManual = false`** に戻す（＝自動連番へ戻すための唯一の導線。空にした直後に `recalcAutoDates` が走り、直前の日を起点とした自動値が入る）
- **UI**: index 1 以降の日で `dateManual === false` かつ日付が非空（＝自動で入っている）の場合のみ、日付入力の隣に控えめな「自動」ヒントバッジ（`day.dateAutoBadge`。4言語対応）を表示する。手動固定の日・空欄の日・Day 1 では表示しない
- 新規に追加される日（「＋ 日を追加」）は既定 `dateManual: false` で作成されるため、直前の日を起点に自動的に日付が埋まる
- viewOnly では日付入力自体が読み取り専用のため編集不可（既存ガード方針に従う）

### 3. タイムライン（メイン）
- 選択中の日の items を上から順に表示
- **時刻の自動計算**: 日の startTime から各項目の dur を累積して、各カードに「開始〜終了」時刻を表示する。項目を並べ替え・時間変更すると全カードの時刻が即再計算される
- 各カードの表示: カテゴリー色の左ボーダー＋アイコン、名前、滞在時間（分。クリックで編集）、メモ、複製ボタン、削除ボタン
- **複製ボタン（⧉）**: そのカードの完全コピー（id のみ新規発行）を直後に挿入する。move も複製可。複製後は時刻・行程番号が再計算される
- move カードはさらに移動手段のセレクトを表示。手段を変えると distKm から所要時間を自動再計算（distKm が null なら時間は手入力のまま）
- カード内の各フィールドはその場で編集できる（input/contenteditableどちらでも可、実装しやすい方で）

### 3a. 行程番号（地図ピンと対応）

- 各日の move 以外の項目に、上から順に 1, 2, 3… の行程番号を振る（日ごとにリセット）
- タイムラインカードに、地図ピンと同じ見た目の丸番号バッジ（カテゴリー色・白抜き数字）を**大きく目立つように**表示する: カード左端のカテゴリー色ボーダーに半分重なる位置、直径28px程度（地図ピンと同サイズ）、白フチ＋影付き。ひと目で行程順が追えることを最優先にする
- move カードには前後の滞在地の番号で「2 → 3」形式のバッジを表示（前または後の滞在地が無い move では非表示）
- **地図ピンの番号はこの行程番号をそのまま使う**（従来の「座標を持つ項目だけで振り直した連番」は廃止）。座標の無い項目の番号は地図上で欠番になるが、それが正しい挙動（＝まだピンが立っていない場所が分かる）
- 並べ替え・追加・削除で番号は自動的に振り直される（時刻再計算と同タイミング）
- テキスト入出力に番号は含めない（行の順序から自明のため）

### 3b. Google Maps リンク（move 以外のカテゴリー）

各滞在地カードに Google Maps の共有リンクを貼れる。さらに未入力でも自動リンクを表示する:

- カード内に小さな「📍 Google Maps URL」入力欄を置く（空でも表示。コンパクトに）
- 有効なリンクが入っているときは、カードの名前の横に 🔗 アイコンのリンクを表示し、クリックで Google Maps を新しいタブで開く（`rel="noopener"`）
- **自動リンク**: gmap が未入力の項目でも 🔗 を表示する。リンク先は描画時に導出（trip には保存しない）:
  - 座標があれば `https://www.google.com/maps/search/?api=1&query=<lat>,<lon>`
  - なければ `https://www.google.com/maps/search/?api=1&query=<loc||name をURLエンコード>`（名前も空なら 🔗 非表示）
  - 手動入力リンク（gmap）がある場合はそちらを優先
- **リンク→名前の自動補完**: gmap 貼り付け時、URLの `/maps/place/<場所名>/` 部分から場所名を抽出する（URLデコードし `+` を空白に置換）。項目の名前が空の場合のみ抽出した場所名を自動入力し、トーストで通知する（既に名前がある場合は上書きしない）。短縮リンクや place を含まないURLでは何もしない
- **URLからの座標抽出**（貼り付け/変更時に即実行）。優先順:
  1. `!3d<lat>!4d<lon>`（場所ピンの正確な座標。data= ブロック内）
  2. `q=<lat>,<lon>` または `query=<lat>,<lon>`（クエリパラメータ）
  3. `@<lat>,<lon>,<zoom>z`（ビューポート中心。近似）
- 抽出に成功したら item.lat/lon に保存し、地図とルート計算はこの座標を最優先で使う（Nominatim 検索より正確）。トーストで「リンクから位置を取得しました」と通知
- `maps.app.goo.gl` などの短縮リンクはクライアントからは展開できない（CORS）。その場合はリンク自体は保存しつつ「短縮リンクからは位置を取得できません。ブラウザで開いた後のURLを貼ってください」とトーストで案内
- 座標の無効化ルール変更: 名前の編集で lat/lon を null に戻すのは「gmap リンク由来の座標でない場合」のみ。gmap を消したり別URLに変えたら座標を抽出し直す（抽出できなければ null）
- 座標のソースを区別するため `coordSrc: "gmap" | "geo" | null` を item に持つ（gmap=リンク抽出、geo=Nominatim）
- 「📍 地図を更新」「ルートを検討」のジオコーディングは、gmap 由来の座標を持つ項目をスキップする（上書きしない）
- `gmapAuto: boolean` を item に持つ（move 以外）。「📍 地図を更新」が自動入力した gmap リンクなら true、手入力（貼り付け・編集）なら false。詳細は 6b の「Google Maps リンクの自動入力」を参照

### 3c. スポット名の多言語表示

スポット名はユーザー入力（通常は日本語）だが、言語切替時に OpenStreetMap の多言語名で表示を補う:

- データ: **全カテゴリー（move含む）** の項目に `names: { en?: string|null, zh?: string|null, th?: string|null, ja?: string|null }` を追加（取得済みの言語のみキーを持つ。null =「取得を試みたが無かった」の記録で再取得しない）。共有リンク・localStorage に含まれ、共有相手にもそのまま表示される
- 取得タイミング: 言語切替時に、全日の「名前があり `names[新言語]` が未取得の項目」（move も対象）を対象に、Nominatim の namedetails 付き検索を逐次実行:
  - `https://nominatim.openstreetmap.org/search?format=json&limit=1&namedetails=1&q=<loc||name>`（座標がある項目は `&bounded=1&viewbox=<±0.02度>` で周辺に限定し同名別施設への誤ヒットを防ぐ）
  - **逆ジオコーディング（座標→名前）は使わない**: その地点にたまたま重なる別の施設名を拾うため（実測で上野公園の座標→「正岡子規記念球場」等の誤りが発生）
  - 1リクエストのレスポンスの `namedetails` から `name:ja` / `name:en` / `name:zh`（`zh-Hans`/`zh-CN` フォールバック）/ `name:th` を**全言語一括で** names に保存する。タグが無い言語は null（=取得済み・名前なし）
  - 既存のレート制限（1.1秒間隔）と同じ仕組みを共用。実行中は進捗トースト（「スポット名を取得中… n/N」）。完了後に render()
  - エラー時は保存せず次回再試行。UI言語が ja のときも同様に動作する（元名が外国語で入力されているケースに対応）
- 表示ルール（表示言語を `L` とする）:
  - `names[L]` があり、かつ元の name と異なる場合のみ、カードの名前入力欄の下に小さく「🌐 <names[L]>」を表示（入力欄自体は常に元の name。編集の正は変えない）
  - move カードのタイトル「A → B」は描画時に前後スポットの `names[L] || name` から組み立てて表示するのを**常に優先**する（保存されている name 文字列は変更しない）。**前後スポットの両方が特定できない move**（ユーザーが手入力した名前の move や、隣接関係が壊れている move）は、この組み立てができないため、代わりに move 自身の `names[L]`（下記フォールバックで取得したもの）が元の name と異なればそれを 🌐 ヒントに表示する。それも無ければ保存名のまま
  - 地図ピンのポップアップの名前も `names[L] || name`
  - テキスト入出力・共有リンク生成では元の name を使う（names はJSONにそのまま含まれる）
- 名前を編集したら、その項目の `names` は全消去する（古い翻訳が残らないように）
- **APIキー設定時の追加フォールバック**（OSM で対象言語の名前が取れなかったスポット向け）:
  1. **Places Text Search を対象言語で照会**: `languageCode: <L>` を付けて `places.displayName` を取得（FieldMask に places.displayName,places.location）。locationBias はスポット自身の座標（radius 1000）、無ければ近隣アンカー（radius 50000）。「ドイステープ」→ Doi Suthep / ดอยสุเทพ / 素贴山 のような正式多言語名が取れる
  2. それも失敗したら **Cloud Translation API v2 で機械翻訳**: `POST https://translation.googleapis.com/language/translate/v2`（ヘッダー `X-Goog-Api-Key`）body `{q: <name>, target: <L>, format: "text"}` → `data.translations[0].translatedText`。「ホテル周辺の喫茶店」のような自由文はこれで翻訳される
  - 取得順: OSM namedetails（全言語一括・1.1秒制限）→ Places（対象言語のみ・制限なし）→ Translation（同）。成功した時点で names[L] に保存して終了
  - **move は OSM・Places を両方スキップし、Translation のみを使う**（move は場所ではないため位置検索系APIが無意味で、無駄なAPI呼び出し・レート制限消費になるため。クエリは `item.name`）
  - スキップ条件の変更: names[L] が**非空文字列のときだけ**取得済みとしてスキップする（null はキー追加後に再試行できるようにする）。OSM ステップ自体は「names にいずれかのキーがある」場合スキップ（OSM は全言語一括保存のため。move はそもそも呼ばない）
  - Places/Translation の 403/400 は実行ごとに1回だけ案内トースト（それぞれのAPI名で）。未設定でも静かにフォールバック
- OSM・Places・Translation すべて使えない場合は元の名前のまま（それが正しいフォールバック）
- **メモの多言語表示（3c 追記）**: move を含む全カテゴリーの項目に `noteNames: { en?: string|null, zh?: string|null, th?: string|null, ja?: string|null }` を追加（`names` と同じ構造・同じスキップ条件）。言語切替時、上記の名前取得と同じ逐次処理の中で「メモが非空で `noteNames[新言語]` が未取得の項目」も対象にする。メモは自由文で場所ではないため **OSM・Places は使わず Cloud Translation API のみ**を使う。表示は名前の 🌐 ヒントと同じ見た目で、メモ欄の下に小さく「🌐 <noteNames[L]>」を表示する（メモ入力欄自体は常に元の note。編集の正は変えない）。メモを編集したらその項目の `noteNames` は全消去する。`notePriv: true` の項目は `sanitizeTripForPublic` が `note` を空にするのと同様に `noteNames` も空オブジェクトにする（翻訳文からメモの内容が公開コピーに漏れないようにするため）。CSVには含めない（言語切替時に再取得できるため。8参照）

### 4. 項目の追加
- タイムライン下部に追加フォーム: カテゴリー4択（色付きボタン）、名前、滞在時間(分)、メモ、追加ボタン
- Enter でも追加できる

### 5. ドラッグ&ドロップ並べ替え
- カードをドラッグして同一日内で並べ替えできる
- **Pointer Events で実装**（HTML5 DnD はスマホ非対応のため不可）。ドラッグ中は元位置を半透明にし、ドロップ位置にインジケータ線を表示
- ドラッグハンドル（⠿）をカード左端に置き、ハンドルからのみドラッグ開始（誤操作防止）
- ドロップ後に時刻を再計算

### 6. 「移動手段の選択とルートを検討」ボタン
タイムライン上部に目立つボタンとして配置。押すと選択中の日について:

1. **その日の `auto: true` の move 項目を一旦すべて削除する**（並べ替え等で前後関係が変わり不要になった自動移動を残さないため。ユーザーが手動追加した move は削除しない）。その後、move 以外の連続する項目ペアを列挙する（間に手動 move が挟まっているペアは生成対象外）
2. 各項目の `loc || name` を Nominatim でジオコーディング:
   - `https://nominatim.openstreetmap.org/search?format=json&limit=1&q=<encoded>`
   - **リクエスト間隔は必ず1.1秒以上あける**（利用規約）。進捗をボタン上に「検索中… 2/5」のように表示
   - 結果は `localStorage`（キー `tabi-geo-cache`）にキャッシュし、同じ地名は再問い合わせしない
3. 2点間の直線距離をハバースイン公式で計算し、**実移動距離 ≒ 直線距離 × 1.3** として補正
4. 距離から移動手段と所要時間を推定:

| 距離(補正後) | 手段 | 計算式（分） |
|---|---|---|
| < 1.5km | walk 徒歩 | dist / 4.5km/h × 60 |
| 1.5〜30km | train 電車 | dist / 25km/h × 60 + 12（待ち・乗換） |
| 30〜200km | train 電車(特急) | dist / 70km/h × 60 + 25 |
| 200〜700km | shinkansen 新幹線 | dist / 180km/h × 60 + 40 |
| ≥ 700km | plane 飛行機 | dist / 700km/h × 60 + 150（空港手続込） |

   5分単位に丸める。ユーザーが move カードの手段セレクトを変えたら、上表の該当手段の式（bus は 18km/h+15, car は 35km/h+5, ferry は 20km/h+20, other は再計算なし）で distKm から再計算する。
5. ペアの間に `auto: true` の move 項目を挿入（既存の auto な move は上書き更新、ユーザーが手で追加した move は触らない）。move の name は「浅草寺 → 上野」のような形式
6. **場所を解決できないスポットの扱い（近隣アンカー概算）**:
   - ジオコーディング（Routes API利用時は住所解決）に失敗したスポットは、「同じ日の直前の解決済みスポットと同じ位置」とみなして概算する（直前に無ければ直後の解決済みスポット）。例:「ホテル→ホテル周辺の喫茶店」は徒歩5分の概算、「ホテル周辺の喫茶店→深大寺」は「ホテル→深大寺」として計算
   - アンカーで概算した move には `approx: true` を保存し、カードに「⚠概算」バッジ（ツールチップで理由説明）を常時表示する。スポット自体の lat/lon には保存しない（地図ピンは欠番のまま＝未確定と分かる）
   - アンカーも見つからない場合（その日に解決済みスポットが1つも無い等）は、dur=0・mode=other・`unresolved: true` の move を挿入し、カードを警告スタイル（淡い警告色＋「⚠ 場所を特定できませんでした」の常時表示）にする。時間入力欄は 0 ではなく空欄＋「時間を入力」プレースホルダにする
   - `approx`/`unresolved` は自動 move の再生成時に毎回リセット・再判定される
   - 従来どおり、完了後にまとめて「見つからなかった場所: ○○」トーストも表示する
7. ジオコーディングに成功した滞在地は、その座標を項目の `lat` / `lon` に保存する（`coordSrc: "geo"`。地図表示・共有で再利用するため）。gmap リンク由来の座標を既に持つ項目は Nominatim を呼ばずその座標を使う

### 6a. 移動カードの「Google Maps でルートを開く」リンク

各 move カードに、区間の前後の滞在地を入れた Google Maps 経路検索リンクを表示する（Googleが計算する実際の所要時間をワンタップで確認し、概算値を手で直せるようにするため）:

- URL: `https://www.google.com/maps/dir/?api=1&origin=<o>&destination=<d>&travelmode=<m>`
  - origin/destination: 前後の滞在地が座標を持てば `lat,lon`、なければ名前（`loc || name`）を URL エンコードして使う
  - travelmode 変換: walk→walking / car→driving / train・bus・shinkansen・ferry→transit / plane・other→パラメータ省略
- move カード内に 🗺「Google Mapsで確認」リンク（新しいタブ、`rel="noopener"`）として表示。前後の滞在地が特定できない move（日の先頭/末尾にある等）では非表示
- 移動手段セレクトを変えたらリンクの travelmode も追随

### 6b. 地図表示（Leaflet）
選択中の日の滞在地を地図で確認できる。

- **ライブラリ**: 同梱の Leaflet 1.9.4（`vendor/leaflet/`）。タイルは `https://tile.openstreetmap.org/{z}/{x}/{y}.png`、attribution は `© OpenStreetMap contributors` を必ず表示
- **レイアウト**:
  - PC幅(≥960px): タイムラインの右横に地図カラムを並べる2カラム。地図は `position: sticky` でスクロール追従
  - モバイル: タイムライン上部に開閉式（「🗺 地図」トグルボタン）。初期状態は閉。開閉状態は localStorage に記憶
- **マーカー**: move 以外の項目のうち `lat`/`lon` を持つものに、訪問順の番号（1,2,3…）入りの丸ピンを表示。ピンの色はカテゴリー色。クリックでポップアップ（名前＋開始〜終了時刻）
  - Leaflet 標準の画像マーカーではなく `L.divIcon` で番号付きの丸ピンを描く（かわいい見た目・カテゴリー色対応のため）
- **経路線**: マーカーを訪問順に結ぶ破線の Polyline（1色、半透明）
- **表示範囲**: マーカー全体が収まるよう `fitBounds`（パディングあり）。マーカーが1つなら zoom 15 で center。0個なら日本全体（[36.5, 138], zoom 5）を表示し「座標のある場所がありません」のオーバーレイメッセージ
- **「📍 地図を更新」ボタン**（地図パネル内）: 座標未取得の滞在地を Nominatim でジオコーディングして `lat`/`lon` に保存し、地図を再描画する。レート制限（1.1秒間隔）・キャッシュ・進捗表示・失敗トーストはルート検討ボタンと同じ仕組みを共用
  - **Google Maps リンクの自動入力**: このボタンの実行後、座標が確定している項目（move 以外。今回新たに解決した項目だけでなく、実行前から既に座標を持っていた項目も対象）のうち `gmap` が空のものに、`https://www.google.com/maps/search/?api=1&query=<lat>,<lon>` を `item.gmap` に保存し `gmapAuto: true` を付ける（手動でリンクを貼った項目は上書きしない）。ジオコーディング対象が1件も無い場合でもこの自動入力だけは実行する
  - `gmapAuto` が true の項目で名前を編集し座標がクリアされる際は、その `gmap`（自動入力されたもの）も一緒にクリアし `gmapAuto` を false に戻す（3b の座標無効化ルールと連動。手動で貼ったリンクは対象外）
  - `gmap` 欄を手動編集したら（貼り付け・変更・削除いずれも）`gmapAuto` は false になる
- **再描画タイミング**: 日の切替、項目の追加/削除/並べ替え、名前・loc の変更、ルート検討完了後。編集のたびに地図を作り直すのではなく、マーカー/線のレイヤーのみ更新する
- 名前や loc が変更されたら、その項目の `lat`/`lon` は無効化（null に戻す）する
- オフライン等でタイルが読めなくても、アプリ本体の動作は影響を受けないこと

### 6c. Google Routes API による正確な移動時間（APIキー設定時）

ユーザーが自分の Google Maps Platform API キーを設定すると、ルート検討の移動時間が距離ベースの概算から Google の実計算値に切り替わる。

**設定UI**:
- ヘッダーに ⚙️ ボタン → 設定モーダル: 「Google Maps APIキー」パスワード型入力欄＋保存/削除ボタン
- キーは `localStorage`（キー `tabi-gmaps-key`）のみに保存。**trip オブジェクトに入れない＝共有リンク・テキスト出力に絶対に含めない**
- モーダル内に注意書き: Routes API の有効化が必要なこと、従量課金であること、HTTPリファラー制限を推奨すること（4言語）

**ルート検討ボタンの動作変更**（キーがあるとき）:
1. 各区間について `https://routes.googleapis.com/directions/v2:computeRoutes` に POST:
   - ヘッダー: `Content-Type: application/json`, `X-Goog-Api-Key: <key>`, `X-Goog-FieldMask: routes.duration,routes.distanceMeters,routes.legs.startLocation,routes.legs.endLocation`
   - origin/destination: 座標があれば `{location:{latLng:{latitude,longitude}}}`、なければ `{address:"<loc||name>"}`（Google側でジオコーディングされるため Nominatim 不要）
   - travelMode: まず TRANSIT で照会。返ってきた distanceMeters が 1.5km 未満なら WALK で照会し直してそちらを採用。TRANSIT でルートが見つからない場合（海外の離島等）は DRIVE で再試行
2. 成功したら: dur = duration（秒→分、5分丸め）、distKm = distanceMeters/1000、**モード表記は実際に使われた travelMode を反映する**: WALK→walk / DRIVE→car / TRANSIT→距離で train（<200km）・shinkansen（≥200km）。距離だけでモードを決めてはならない（例: タイなど鉄道網の無い地域では TRANSIT が失敗して DRIVE で再照会されるが、その結果を「電車」と表示するのは誤り。「車」と表示する）
3. legs の startLocation/endLocation の latLng を前後の滞在地の lat/lon に保存（coordSrc:"geo" 扱い。gmap 由来の座標は上書きしない）→ 地図ピンにも使われる
4. move カードの手段セレクト変更時: キーがあれば該当モード（walk→WALK / car→DRIVE / train・bus・shinkansen・ferry→TRANSIT）で再照会して dur を更新（照会中はセレクト無効化）。plane/other は従来の式（plane）/変更なし(other)
5. **キャッシュ**: `localStorage`（キー `tabi-routes-cache`）に `origin|dest|mode` → {durMin, distKm} を保存し、同一照会を繰り返さない（課金抑制）。キャッシュヒット時はAPIを呼ばない
6. APIエラー（4xx/5xx・ネットワーク）時はトーストで通知し、その区間は従来の Nominatim＋距離式にフォールバック。キー無効(403)が返ったら「APIキーを確認してください」トースト
7. transit の照会には出発時刻が影響するため、`departureTime` にその区間の開始予定時刻（日の日付＋累積時刻。日付が過去なら翌日の同時刻）を ISO8601 で渡す

**キーがないとき**: 従来どおり Nominatim ＋距離ベース概算（変更なし）。

### 6d. スポット位置解決の統一チェーン（Places API フォールバック）

「ドイステープ」のような海外スポットの日本語表記は Nominatim にも Routes API の住所解決にも載っていないことが多い。
ルート検討・地図更新で使うスポットの位置解決を次の優先順に統一する:

1. 項目が既に座標を持つ（gmap リンク由来含む）→ そのまま使う
2. Nominatim 検索（既存。キャッシュ・1.1秒制限あり）
3. **APIキーがあれば Google Places API (New) の Text Search**:
   - `POST https://places.googleapis.com/v1/places:searchText`
   - ヘッダー: `Content-Type: application/json`, `X-Goog-Api-Key: <key>`, `X-Goog-FieldMask: places.location,places.displayName`
   - ボディ: `{ "textQuery": "<loc||name>", "languageCode": "<現在のUI言語>", "pageSize": 1 }`
   - **locationBias（重要）**: 同じ日に座標を持つスポットがあれば、最も近い位置関係のもの（直前優先）を中心に `locationBias: { circle: { center: {latitude, longitude}, radius: 50000 } }` を付ける。チェーン店名（例: セントラルフェスティバル＝タイ全土にあるモール）で別都市の店舗にヒットするのを防ぐため。キャッシュキーにはバイアス中心の粗い位置（小数1桁）も含める
   - `places[0].location.latitude/longitude` を採用。成功したら項目の lat/lon に保存（coordSrc:"geo"）し、`tabi-geo-cache` にも `places:<query>` キーでキャッシュする
   - 403/400（API未有効・キー制限）は「Places API が有効化されていません」系のトーストを1回だけ表示し、以降このチェーンをその実行内ではスキップ
4. それでも不明 → 既存の近隣アンカー概算 / unresolved 警告

設定モーダルの注意書きに「Places API (New) を有効化すると海外スポット等の名前解決が強化される（任意）」を追記する（4言語）。

### 6e. しおりデータの多言語タイトル

- `trip.titles = { ja?: string, en?: string, zh?: string, th?: string }` を追加。表示は `titles[現在言語] || title`
- ヘッダーのタイトルを編集すると `titles[現在言語]` に保存する（`title` は後方互換のフォールバックとして、ja 編集時のみ同期更新）
  - **ただし blur 時に表示されていた値（`titles[現在言語] || title`）から変わっていない場合は何も保存しない。** 変更が無いのに現在の表示文字列（他言語の場合、翻訳前のベースタイトルがそのまま出ていることがある）を `titles[現在言語]` に書き込んでしまうと、下記の自動翻訳のスキップ条件に誤って引っかかり、その言語のタイトルが永久に翻訳されなくなるバグを防ぐため
- **自動翻訳（APIキー設定時）**: 言語切替時に `titles[新言語]` が無い（または空）場合、Cloud Translation API v2 でベースタイトル（`titles[ja] || title`）を新言語に翻訳して `titles[新言語]` に保存する。手動編集はいつでも上書き可能。API未有効・エラー時は静かにベースタイトルのまま（トーストはスポット名側の仕組みと共通で実行ごと1回）
  - **スキップ条件の変更**: `titles[新言語]` が非空文字列で、かつベースタイトル（`titles[ja] || title`）と異なる場合のみ「翻訳済み」とみなしてスキップする。`titles[新言語] === ベースタイトル` の場合は「未翻訳のまま」とみなして翻訳を実行し、成功したら上書きする（`新言語 === "ja"` はこの処理の対象外＝常にスキップ）
- サンプルしおりは4言語分のタイトルをプリセットする（東京旅行 / Tokyo Trip / 东京之旅 / ทริปโตเกียว）
- 共有リンク・テキスト入出力のタイトルは `titles[現在言語] || title` を出力。テキスト読込では現在言語の titles に取り込む

### 7. 共有
- 「共有」ボタン → モーダル: trip 全体ではなく **`sanitizeTripForPublic(trip)` の結果（非公開マークを除去した公開コピー）** を JSON→UTF-8→base64url にして `location.origin+pathname+"#d=..."` のリンクを生成、「コピー」ボタンでクリップボードへ（詳細は 14 参照）
- モーダルに「このリンクには非公開項目は含まれません。」の注記、および「公開時の見え方を確認」ボタン（公開プレビューモーダルを開く。14参照）を表示する
- ページ読込時に `#d=` があればパースし、「共有されたしおりを開きますか？（現在のデータは上書きされます）」と確認してから読み込む。読み込んだらハッシュは消す
- base64url は UTF-8 対応必須: `btoa(String.fromCharCode(...new TextEncoder().encode(json)))` 方式＋ `+/=` を `-_`(パディング除去) に置換

### 8. CSV 入出力
「CSV入出力」ボタン → モーダルに textarea。現在の trip を下記のCSV形式で表示し、編集して「読み込む」を押すと現在のしおりの days を置き換える（確認ダイアログあり）。ボタン: コピー / 読み込む / CSVダウンロード / ファイルを開く。

```
title,東京旅行
day,date,start,category,mode,name,minutes,note,gmap
1,2026-07-20,09:00,観光,,浅草寺,90,雷門で写真,https://www.google.com/maps/...
1,2026-07-20,09:00,移動,車,浅草寺 → 上野,25,,
1,2026-07-20,09:00,食事,,上野でランチ,60,,
```

- ヘッダー行は英語キー固定（`day,date,start,category,mode,name,minutes,note,gmap`）。インポート時はヘッダー行必須で、列順はヘッダーに従う
- day/date/start は行ごとに繰り返す（同じ day の date/start は最初の行の値を採用）
- category・mode は現在の言語で出力し、インポートは4言語いずれの表記も受け付ける（従来のエイリアス表を流用）。mode は move 以外では空
- RFC4180 準拠の引用符処理（`,` `"` 改行を含むフィールドは `"..."` で囲み、`"` は `""` にエスケープ）を出力・パース両方で行う
- 「CSVダウンロード」は Blob で .csv 保存（**UTF-8 BOM 付き**。Excelでの文字化け防止）。「ファイルを開く」は input type=file で .csv を textarea に読み込む（読み込むボタンは従来どおり）
- パースできない行は「N行目を読み込めませんでした」と警告（続行）。gmap 列は座標抽出も行う
- 持ち物リスト・やることリストのCSV入出力（`list,text,done` の第2テーブル）は 10 を参照
- 行程テーブルのヘッダーに任意列 `private`（1/0）・`notePrivate`（1/0）、チェックリストテーブルのヘッダーに任意列 `private`（1/0）を追加できる（後方互換・詳細は 14 参照）。**CSVエクスポートは自分用の完全バックアップのため、非公開項目も含めた完全データを出力する**（サニタイズ対象外）
- **CSVタイトル行（8 追記）**: 本文の1行目に `title,<タイトル>` の2フィールド行を出力する（値は `tripDisplayTitle(trip)`。行程テーブルのヘッダー行 `day,date,...` はその次の行から始まる）。CSVにタイトル情報が一切無く、新規しおりへCSVをインポートしてもタイトルが引き継がれず既定値（「新しい旅行」等）のまま残ってしまう問題への対応
  - インポート時: 1行目が `title` で始まる2フィールド行ならタイトルとして取り込み、`titles[現在のUI言語]` に設定する（現在のUI言語が `ja` のときは `title` にも同じ値を設定する。6e の「ja が基準」という設計に合わせるため）。取り込んだ後、2行目以降を従来どおり解析する
  - タイトル値が空文字列、または行が壊れている（2フィールドちょうどでない）場合は何もしない（既存のタイトルを保持する。クラッシュしない）
  - **後方互換**: 1行目が `day,date,...` で始まる旧CSV（タイトル行なし）はタイトル行が無いものとして従来どおり読み込み、タイトルは変更しない
  - メモの翻訳キャッシュ（`noteNames`。3c 参照）はCSVに含めない。言語切替のたびに再取得できるため出力する必要が無い

### 9. 複数しおりの管理

複数の旅行（しおり）を保存・切替できる:

- ストレージ v2: `localStorage` キー `tabi-shiori-v2` = `{ currentId: string, trips: [{ id: string, data: <trip>, archived: boolean }] }`（`archived` は 11 参照）
  - 起動時に旧キー `tabi-shiori-v1` があれば最初のしおりとして移行し、旧キーは残さず削除
- ヘッダーに 📚 しおり一覧ボタン → モーダル:
  - しおりのリスト（タイトル＝`titles[言語]||title`、日数、現在のしおりをハイライト）。クリックで切替。既定ではアーカイブされていないしおりのみ表示（11参照）
  - 「＋ 新しいしおり」: 空の Day 1 だけを持つ新しいしおりを作成して切替（タイトルは現在言語のデフォルト「新しい旅行」等、titles 4言語プリセット）
  - 各しおりに削除ボタン（確認ダイアログ。アーカイブされていないしおりが1つだけのときは削除不可。11参照）
- 共有リンク（#d=）を開いたときは**上書きではなく新しいしおりとして追加**して切替する（確認ダイアログの文言も「新しいしおりとして開きますか？」に変更）
- 自動保存・言語設定は従来どおり（言語 trip.lang はしおりごと）

### 10. 持ち物リスト・やることリスト

タイムラインの項目追加フォームの下に、しおり単位（日ごとではない）の持ち物リストとやることリストを置く。チェックボックス付きのシンプルなリストで、荷造りや旅行前の準備を家族・同行者と共有できる。

- **データ**: trip に `packing: [{id, text, done}]` と `todos: [{id, text, done}]` を追加（しおり単位）。`normalizeTrip` で防御的に正規化する（text は文字列、done は boolean、id が無ければ `genId()`）。localStorage・共有リンクに含まれる（＝共有相手にも見える／持ち物の分担に使える）
- **配置**: タイムラインの項目追加フォーム（4）の下に、2つのセクションを縦に並べる。PC幅（目安 ≥700px）では横2カラム、モバイルは縦積み。見出しは「🧳 持ち物リスト」「✅ やることリスト」（4言語対応）
- **各セクション**:
  - チェックボックス＋テキスト（インライン編集可）＋削除ボタンの行が、項目ごとに白い角丸カードとして縦に並ぶ
  - 下部に「＋ 追加」の入力欄（Enterでも追加できる）
  - チェック済みの項目は打ち消し線＋淡色表示にする
  - 見出し横に「3/8」形式の進捗カウント（完了数/全体数）を表示する
  - 各行をドラッグして並べ替えられる（10a参照）
  - 空のときは薄いプレースホルダ文を表示する（例:「持ち物を追加しましょう」「やることを追加しましょう」）
- **CSV入出力（8 の拡張）**: 「CSV入出力」モーダルで出力するCSVに、既存の行程CSVの後に空行を1行挟んだ第2テーブルを追加する。ヘッダーは `list,text,done` 固定（英語キー）。list列は現在言語での「持ち物」/「やること」表記、done列は `1`/`0`。RFC4180の引用符処理（`csvEscapeField` / `parseCsvRows`）は行程CSVと共通の関数を流用する
  - インポート時、行程CSVの後の最初の空行以降に `list,text,done` ヘッダーの表があれば、それを解析して packing/todos を**置換**する
  - 第2テーブルが無い（区切りが見つからない、またはヘッダーが一致しない）場合は、後方互換のため既存の packing/todos を**そのまま維持**する（旧形式のCSVを読み込んでもエラーにならない）
  - **手編集されたCSVへの耐性**（重要）: 出力したCSVを手作業やExcelで編集して読み込み直すのは通常の使い方であり、その際に空行の形は保たれない。そのため区切りの判定は次のように緩くする:
    - 空行は「フィールドが1つで空文字」に限定せず、**全フィールドが空白のみの行**を空行とみなす（`isBlankCsvRow`）。Excelでの保存は空行を `,,,,,,` に変え、手書きでは空白が混入しうるため
    - 空行が失われている場合に備え、**`list,text,done` を含むヘッダー行そのもの**も区切りとして探す（`isChecklistHeaderRow`）。この場合ヘッダー行の直前までが行程テーブル
    - 行の読み飛ばし判定にも同じ空行判定を使う（カンマだけの行を「列数不足」として警告しない）
  - list列は4言語いずれの表記も受け付ける（持ち物/Packing/行李/ของที่ต้องเตรียม、やること/To-do/待办/สิ่งที่ต้องทำ）。既存のカテゴリー・移動手段のエイリアス解決と同じ仕組みを使う
- packing/todos は trip オブジェクト直下（しおり単位）にあるため、既存のタイムライン項目の複製ボタン（⧉、3参照）には影響しない。共有・しおり切替とは自然に整合する: しおりを切替・共有すればそのしおりの packing/todos も一緒に移動する（しおりごとに独立したリスト）
- **準備リストへのクイックアクセス**: 行程が長いとタイムライン下の持ち物・やることリストまでスクロールするのが大変なため、ヘッダーの 📚 しおり一覧ボタンの隣に 🧳 ボタン（aria-label「持ち物・やること」4言語）を置き、ワンタップで**準備モーダル**を開けるようにする
  - 準備モーダルの中身は、上記のタイムライン下セクションと**まったく同じ描画関数**を使い回す（チェック・インライン編集・削除・追加・進捗カウントいずれも同一挙動）。データはしおり単位で共有しているため、モーダルとタイムライン下のどちらで編集しても、もう片方に即座に反映される
  - タイムライン下のセクションは引き続きそのまま残す（クイックアクセスは追加の入口であり、置き換えではない）
  - 未完了件数（持ち物・やること合算）が1件以上あるとき、ヘッダーの 🧳 ボタンに小さな数字バッジを表示する。0件のときはバッジを隠す

#### 10a. 持ち物・やることリストの並べ替え

持ち物リスト・やることリストの各行をドラッグして並べ替えられる。

- 各行の左端にドラッグハンドル（⠿）を置き、**ハンドルからのみドラッグ開始**する（チェックボックス・テキスト編集を誤操作で邪魔しないため）
- **main（タイムライン下）と prepModal（🧳）の両方で動作する**。`renderChecklistSection` が両方に描画する既存の構造にそのまま乗せているため、片方で並べ替えるともう片方にも即座に反映される（renderChecklistSection の既存パターン）
- 並べ替え確定後は対象配列（`trip.packing` または `trip.todos`）の順序を更新し、`saveState()` → `renderChecklistSection(kind)` で再描画する
- viewOnly ではドラッグハンドル自体を表示しない（既存の `.view-only-mode` スコープCSSガードに追加）。JS側も操作関数の冒頭で `if (viewOnly) return;` する
- **実装方針**: タイムラインのドラッグ実装（5、Pointer Events・`dragState`・ドロップインジケータ）と**同じ設計思想**（Pointer Events + ハンドルからのみ開始 + ドロップインジケータ表示）を踏襲するが、対象配列がタイムラインの `day.items` 1つに対し持ち物・やることの2種類（`trip.packing`/`trip.todos`）ある点、描画先が `el.timeline` 1箇所に対し main/prepModal の2箇所ある点が異なるため、**別関数（`checklistDragState`・`onChecklistDragPointerDown`系）として実装する**（HTML5 Drag & Drop APIはスマホ非対応のため使わない、という方針は共通）
- CSV（8・10）・共有リンク／公開コピー（7・14・16）はいずれも配列順がそのまま出力・保存されるため、この並べ替え機能のために追加の対応は不要（並べ替え後にエクスポート・共有すれば新しい順序がそのまま反映される）

### 11. しおりのアーカイブ

終わった旅行を一覧から隠せるようにする。

- **データ**: ストレージ v2 の trips エントリ（`{ id, data }`）に `archived: boolean` を追加する（`{ id, data, archived }`）。trip データ本体（`data`）ではなく trips エントリ側に持たせる。`loadStoreV2` の読み込み時・v1移行時に防御的に正規化する（既定 `false`）
- **しおり一覧モーダル**:
  - 通常はアーカイブされていないしおりのみを一覧表示する
  - リストの下に「アーカイブ済みを見る（n）」トグルを表示する（n はアーカイブ件数。0件のときはトグル自体を非表示）。トグルONでアーカイブ済みしおりのセクションを一覧の下に表示する
  - 各しおり行に 📦 アーカイブボタンを置く（アーカイブ済みの行では ↩ 復元ボタンになる）。確認ダイアログは無しで即実行し、完了をトーストで知らせる（「アーカイブしました」「アーカイブから復元しました」、4言語）
  - **現在編集中のしおりをアーカイブした場合**は、アーカイブされていない他のしおりへ自動的に切り替える。他に切り替え先が無い場合はアーカイブを実行せず、トーストで「最後のしおりはアーカイブできません」と伝える
  - アーカイブ済みしおりの削除は可能（既存の削除ボタン・確認ダイアログをそのまま使う）
- **削除ガードの整合**: 「最後の1つは削除できない」という既存ガード（9）を、「アーカイブされていないしおりが1つだけのとき、そのしおりは削除できない」に変更する（アーカイブ済みしおりはこのガードの対象外＝自由に削除できる）
- 共有リンク（#d=）で新規追加されるしおりは常に `archived: false`
- 自動保存（`saveState`）・しおり切替・CSV・共有などの既存機能はアーカイブ状態の影響を受けない（アーカイブは表示上の絞り込みのみ）

### 12. PDF出力（印刷）

外部ライブラリ禁止のため、**印刷用ビュー + `window.print()`**（ブラウザの「PDFとして保存」機能）で実現する。

- ヘッダーに「🖨」ボタン（aria-label「PDF出力」。4言語対応）を配置する
- クリック時の動作:
  1. 専用の静的DOM `#printView` を組み立てて `document.body` の末尾に追加する。**input / select / button は一切使わず、textContent ベースの静的要素のみで描画する**（入力欄の枠線が印刷結果に出るのを避けるため）
  2. `document.body` に `printing` クラスを付与する。`@media print` ＋ `.printing` の組み合わせ時に、`#printView` 以外の要素（ヘッダー・日タブ・タイムライン・地図・追加フォーム・モーダル・トースト。すなわち `body > *:not(#printView)`）をすべて `display: none` にする
  3. `window.print()` を呼び出す
  4. `afterprint` イベント（＋ `window.print()` 呼び出し後のフォールバックタイマー）で `printing` クラスを外し、`#printView` の中身を空にしてDOMから削除する
- `#printView` の内容（**しおり全体・全日分を対象とする。選択中の日だけではない**）:
  - タイトル（`titles[現在言語] || title`）、日付範囲（先頭日と最終日の `date` が両方あれば表示。同一なら1つだけ）
  - 各日ごとに「Day N ｜ 日付 ｜ タイムゾーン名（設定時のみ）」の見出しを置き、その日の全項目を時系列（`getDayTimedItems` の順序どおり）で列挙する。1項目は「開始〜終了時刻 ／ 行程番号 ／ カテゴリーアイコン＋名前（`names[現在言語]` があれば併記）／ 滞在時間 ／ メモ」の形式。move 項目は「移動手段名＋所要時間＋時差表示（13の時差バッジと同じ表記）」に置き換える
  - 持ち物リスト・やることリストは、それぞれの見出しの下にチェック状態を `☑`（完了）/ `☐`（未完了）の文字で表現して列挙する
- 印刷用CSS（`@media print` 内、A4想定）: `@page { margin: 12mm }`。インク節約のため色は薄いグレースケール寄りにし、カテゴリー色は項目の左ボーダーの細線程度に留める（背景色に `print-color-adjust: exact` は使わない）。`break-inside: avoid` を各項目・Day見出しに適用し、ページ境界で分断されないようにする
- **地図はPDF出力に含めない**（Leaflet地図タイルの印刷描画が環境によって不安定なため、意図的に除外する）

### 13. 時差対応

「羽田 08:55 発 → シンガポール 15:00 着（現地時間）」のような、日をまたがない時差の発生を表現する。

- **データ**: `day.tz`（IANAタイムゾーン文字列。既定 ""＝時差計算なし）、move項目に `arriveTz`（IANAタイムゾーン文字列、任意。既定 ""）を持つ。`normalizeTrip` で防御的に正規化する（文字列のみ許容。それ以外の型は既定値にフォールバック）
- **オフセット取得**: `Intl.DateTimeFormat("en-US", { timeZone: tz, timeZoneName: "longOffset" })` の `formatToParts` から `"GMT+09:00"` 形式の文字列を取り出し、分単位の数値に変換するユーティリティ `tzOffsetMinutes(tz, baseDate)` を用意する。基準日 `baseDate` は `day.date`（無ければ現在日時）。無効なタイムゾーン文字列は `try/catch` で捕捉して `null` を返し、時差なし（調整しない）として扱う
- **時刻計算（`getDayTimedItems` の拡張）**: 現在タイムゾーンの変数を `day.tz` で初期化し、各項目の `dur` を分単位で累積する。`arriveTz` が設定された move 項目を通過したとき、`tzOffsetMinutes(arriveTz) - tzOffsetMinutes(現在tz)` をその move 自身の終了時刻（`endMin`）以降に加算し、現在タイムゾーンを `arriveTz` に更新する。既存の「1440分超で `(+n)` を付けて表示する」ロジックはそのまま維持する
  - **重要な制約**: `getDayTimedItems` の戻り値は `day.items` と常に1:1のインデックス対応を保つ（行程番号バッジ・地図ピンの番号・PDF出力ビューがこの対応関係に依存しているため、要素を増減させたり順序を変えたりしない）
- **UI**:
  - 日設定欄（日付・開始時刻の隣）に「🕐 タイムゾーン」セレクトを置く。選択肢は `Intl.supportedValuesOf("timeZone")` が使える環境ではそれを使い、使えない環境向けに主要都市の固定リストへフォールバックする。先頭に「（未設定）」を置く。各選択肢の表示は `"Asia/Tokyo (GMT+09:00)"` のような形式にする
  - move カードに「到着地のタイムゾーン」セレクトを追加する（同じ選択肢一覧＋先頭に「（時差なし）」）。スペースが厳しいため、既存の移動手段セレクトの隣にコンパクトに配置する。`arriveTz` が設定されている move カードには「🕐 -1h」のような時差バッジを表示する
  - `arriveTz` の通過によって時差調整が発生した直後の項目（その move の次の項目）の時刻列にだけ、「(現地時間)」という控えめな注記を1回表示する（以降の項目には毎回表示しない）
- **後方互換**: `tz` / `arriveTz` が未設定のしおりは、従来とまったく同じ時刻計算・表示になる（既存しおりへの影響ゼロ）
- **CSV連携**（8の拡張）: 行程CSVのヘッダーに任意列として `tz`（day列と同様に行ごとに繰り返す）・`arriveTz`（move行のみ）を追加してよい。**これらの列を持たない旧形式のCSVを読み込んでもエラーにならず、`tz`/`arriveTz` は空（時差なし）として扱う**（後方互換必須。列の有無は `colIndex.tz != null` 等で判定し、必須列の判定ロジックには含めない）

### 14. 非公開マークと公開用データ

将来 Google ログイン＋Firestore を導入し（15、後続実装）、「本人だけが読める完全データ」と「誰でも読める公開コピー」の2層に分ける設計を予定している。公開コピーは非公開マークが付いた部分を削除したデータを書き出すため、公開側からは非公開データが物理的に取得できない。本節はその土台となる「マーク付け」と「サニタイズ関数」、および既存のURLハッシュ共有への適用を実装する。

- **データ**:
  - 行程項目（move含む全カテゴリー）に `priv: boolean`（項目まるごと非公開。既定 false）と `notePriv: boolean`（メモのみ非公開。既定 false）を追加する
  - 持ち物リスト・やることリストの各要素（`packing`/`todos`）に `priv: boolean`（既定 false）を追加する
  - `normalizeTrip` / `normalizeChecklist` で防御的に正規化する（boolean 以外の型は既定 false にフォールバック）
  - 新規作成される項目（追加フォーム・サンプルしおり・空のしおり・ルート検討ボタンが生成する move・チェックリスト追加）は既定 `priv: false` / `notePriv: false` で作成される

- **サニタイズ関数 `sanitizeTripForPublic(tripData)`**（今回の中核）:
  - 引数を変更しない純粋関数。深いコピーを作ってから加工し、加工後のコピーを返す
  - `priv: true` の行程項目を、その項目が属する日の `items` から削除する
  - `notePriv: true` の項目は削除せず `note` を空文字にする（項目自体は残す）
  - `priv: true` の `packing`/`todos` 要素を配列から削除する
  - **auto move の整合性**: 非公開項目の削除で前後関係が変わる自動生成 move（`auto: true`）は、削除前の隣接する滞在地（`findAdjacentStops` 相当のロジックで判定した前後スポット）のいずれかが削除された場合、その move も一緒に削除する。ユーザーが手動追加した move（`auto: false`）は隣接スポットが消えても残す
  - 結果のオブジェクトから `priv` / `notePriv` フラグ自体を取り除く（公開データに非公開マークの痕跡を残さない）
  - 除外件数（行程項目・持ち物・やること合計）が必要な画面は、加工前後の要素数の差分から算出する（関数自体は件数を返さない）

- **UI（タイムラインカード）**:
  - 各行程カードに 🔓/🔒 のトグルボタンを、複製・削除ボタンと並べて配置する。ONで `item.priv = true`、カードに淡い「🔒 非公開」バッジを表示し、背景色をわずかに沈ませる
  - メモ入力欄の横に小さな 🔒 トグル（`item.notePriv`）を置く。ONのときはメモ欄の下に「共有相手には表示されません」という控えめなヒントを表示する（4言語）
  - いずれのトグルも、変更のたびに `saveState()` → `render()` する（他の項目編集と同じ即時保存方式）

- **UI（持ち物・やることリスト）**:
  - 各行に 🔓/🔒 トグルボタンを削除ボタンの隣に置く。ONで `it.priv = true` とし、行に🔒アイコンを表示する（トグルボタン自体がその表示を兼ねる）

- **公開プレビュー**:
  - 共有モーダルに「公開時の見え方を確認」ボタンを追加する。押すと `sanitizeTripForPublic(trip)` の結果を専用モーダルに読み取り専用のテキスト一覧として表示する（印刷ビュー用の `buildPrintDaySection` / `buildPrintItemRow` / `buildPrintChecklistSection`（12参照）を流用して DOM を組み立てる。いずれも `textContent` ベースのため XSS対策は既存のまま）
  - 非公開で除外された件数（行程項目・持ち物・やることの合計。加工前後の要素数の差分で算出）を「非公開のため {n} 件を除外しました」の形式で表示する（4言語。0件のときは「非公開の項目はありません」）

- **既存の共有リンク（7）への適用**:
  - `openShareModal` が埋め込むデータを、trip 全体そのままではなく `sanitizeTripForPublic(trip)` の結果に変更する。これにより現時点でも「非公開マークを付けた部分は共有リンクを渡しても見えない」が成立する
  - 共有モーダルに「このリンクには非公開項目は含まれません。」という注記を追加する（4言語）
  - 受信側（`checkSharedHash` → `normalizeTrip`）が読み込むのは常にサニタイズ済みデータ（`priv`/`notePriv` キー自体が存在しない）になるため、共有リンク経由で追加されるしおりの項目は既定値（`priv: false` / `notePriv: false`）で正規化される

- **CSV（8）への適用**:
  - 行程テーブルのヘッダーに任意列 `private`（1/0）・`notePrivate`（1/0）を追加する（`tz`/`arriveTz` と同様、必須列ではなく各項目行の値を持つ）
  - チェックリストテーブル（`list,text,done`）のヘッダーに任意列 `private`（1/0）を追加する
  - **後方互換必須**: これらの列を持たない旧形式のCSVを読み込んでもエラーにならず、`priv`/`notePriv` は列が無い場合 `false` として扱う（列の有無は `tz`/`arriveTz` と同じ判定方法。必須列の判定ロジックには含めない）
  - CSVエクスポートは（自分用の完全バックアップであるため）**非公開項目も含めた完全データ**を出力する。CSVを介したデータのやり取りはサニタイズの対象外である

#### 14拡張: 日単位・リスト全体単位の非公開マーク

項目単位（`priv`）・メモ単位（`notePriv`）・持ち物/やること各項目単位（`priv`）に加え、**日単位**と**持ち物/やることリスト全体単位**の非公開マークを追加する。公開コピーは非公開マークが付いた部分を物理的に削除する方針（14の中核）を維持する。

- **データ**:
  - `day.priv: boolean`（既定 false）— その日を丸ごと非公開にする
  - `trip.packingPriv: boolean` / `trip.todosPriv: boolean`（既定 false）— 持ち物リスト・やることリストをそれぞれ丸ごと非公開にする
  - `normalizeTrip` で防御的に正規化する（boolean 以外の型は既定 false にフォールバック）
  - `createSampleTrip` / `createBlankTripData` / 「＋ 日を追加」ボタンが生成する新しい日は既定 `priv: false` で作成される。しおり作成時の `packingPriv` / `todosPriv` も既定 false

- **`sanitizeTripForPublic` の拡張**:
  - `day.priv === true` の日を `days` 配列から丸ごと削除する（その日に属す項目・move はすべて一緒に消える。日ごと削除のため、既存の「隣接 auto move の削除」ロジックを重ねて適用する必要はない）
  - `packingPriv === true` なら `packing` を空配列に、`todosPriv === true` なら `todos` を空配列にする（リスト全体非公開のときは要素単位の `priv` を個別に見る必要はない）
  - 結果から `priv`（日）・`packingPriv`・`todosPriv` フラグ自体を取り除く（他の priv/notePriv 同様、公開データに痕跡を残さない）
  - 純粋関数のまま（元データは変更しない）
  - **境界条件（全ての日が非公開の場合）**: `days` が空配列になり得る。受け取り側は必ず `normalizeTrip` を通す設計になっており、`normalizeTrip` は空 `days` を1つの空の日（`{ date: "", startTime: "09:00", items: [] }`）にフォールバックする既存の正規化方針があるため、共有リンク（`checkSharedHash`）・公開URL閲覧（`checkPublicHash`）はクラッシュしない。`sanitizeTripForPublic` の結果を `normalizeTrip` を介さず直接 `forEach` するだけの画面（公開プレビューモーダル・公開URL発行時の除外件数計算）も、空配列なら何も描画しない／0件計算になるだけでエラーにはならない

- **除外件数（`countSanitizedExclusions`）**: 加工前後の要素総数の差分で算出する既存の実装のままでよい。日ごと削除された項目・リスト全体非公開で空になった件数も、総数の差分に自動的に反映されるため、追加のロジックは不要

- **UI（日単位）**:
  - 日付・開始時刻・タイムゾーンが並ぶ日設定行（`.day-meta`）に 🔓/🔒 トグルを追加する。ONで `day.priv = true`、「🔒 この日は非公開」バッジを表示する
  - 日タブにも、非公開の日には小さく🔒を表示する（一覧性の確保）
  - viewOnly（公開URL閲覧）モードではこのトグルも無効化される（他の🔒トグル同様、CSSの `.view-only-mode` スコープで非表示にする）

- **UI（リスト単位）**:
  - 持ち物・やること各セクションの見出し行に、進捗カウント（例「3/8」）と並べて 🔓/🔒 トグルを置く。ONならセクション全体を淡い配色にし、「🔒 非公開」バッジを表示する
  - タイムライン下（メイン）・準備モーダル（`prepModal`）の両方に同じトグルを表示する（`renderChecklistSection` が main/prep 両方へ描画する既存の共有ロジックに乗せる）
  - リスト全体が非公開のときも、個々の項目の 🔒 トグルは残す（リスト全体の非公開を解除したときに個別設定が活きるため）
  - viewOnly モードではこのトグルも無効化される

- **CSV（8）への適用（後方互換必須）**:
  - 行程テーブルに任意列 `dayPrivate`（1/0）を追加する。`tz` と同様に行ごとに繰り返し出力し、インポートは同一 day の最初の行の値を採用する
  - チェックリストテーブルに任意列 `listPrivate`（1/0）を追加する。行ごとに出力し、インポートは各 list（持ち物/やること）の最初の行の値を採用する
  - **既知の制限**: リストが空 かつ `listPrivate=true` の場合、そのリストに属す行が1行も存在しないため、CSV往復で `listPrivate` フラグが失われる（既定 false に戻る）。空の非公開リストは実害がないため、この制限は許容する
  - これらの列を持たない旧CSVを読み込んでもエラーにならず、`dayPrivate`/`listPrivate` は列が無い場合 `false` として扱う（既存の `tz`/`private`/`notePrivate` と同じ判定方法）
  - CSVエクスポートは従来どおり自分用の完全データ（非公開項目・非公開の日・非公開リストも含む）を出力する

### 15. Google ログインとクラウド保存

Google アカウントでログインすると、しおりのデータ（本人だけが読める完全データ）を Firestore にも保存し、複数端末間で同期できる。本節は**プライベート層のみ**を扱う。誰でも読める公開コピー（`publicTrips` コレクション・公開URL）は **16** で実装する。**未ログインでも既存の全機能がこれまでどおり使えることが絶対条件**で、ログインはあくまで任意のクラウド同期機能として上乗せする。

- **SDKの同梱**（外部CDN禁止方針の維持）:
  - Firebase JS SDK **compat版**（`firebase-app-compat.js` / `firebase-auth-compat.js` / `firebase-firestore-compat.js`、v12.16.0）を `vendor/firebase/` に同梱する。既存 app.js が非モジュールの `<script src>` 方式のため、ESM版ではなく compat版（UMD、`window.firebase` を公開）を使う
  - `index.html` で `i18n.js` → `leaflet.js` → Firebase SDK 3ファイル → `app.js` の順に読み込む
  - `initFirebase()` は try/catch で包み、`window.firebase` が存在しない・初期化に失敗した場合は `firebaseReady = false` のまま処理を継続する。この場合ログインボタン自体を非表示にし、それ以外は通常どおりローカルのみで完全に動作する（エラートーストは出さない）

- **Firebase設定値**: `firebaseConfig`（apiKey・authDomain・projectId 等）は `app.js` 冒頭に定数 `FIREBASE_CONFIG` として直書きする。これはクライアントに埋め込む前提の公開値であり、実際のアクセス制御は Firestore セキュリティルール側（後述）が担う

- **認証**:
  - ヘッダーに認証ボタン（`#authBtn`）を1つ配置する。未ログイン時は「ログイン」ラベル（4言語）、ログイン時はメールアドレス/表示名の頭文字1文字（無ければ👤）を丸いアバター風に表示する
  - 未ログイン時にクリックすると `signInWithPopup(GoogleAuthProvider)` を実行する。`auth/popup-blocked` エラー時のみ `signInWithRedirect` にフォールバックする
  - ログイン時にクリックするとアカウントモーダル（`#authModal`）を開く。モーダルにはメールアドレスと「ログアウト」ボタン（`signOut()`）を表示する
  - `onAuthStateChanged` でログイン状態を購読し、状態が変わるたびにヘッダー表示・アカウントモーダルを更新する（`renderAuthUI()`）

- **データ構造（Firestore）**: `trips/{cloudId}` ドキュメント
  ```js
  {
    ownerUid: "…",        // request.auth.uid と一致させる
    data: "{...}",         // JSON.stringify したしおり本体（ネスト構造のまま入れず文字列化する）
    title: "東京旅行",      // 一覧表示用（tripDisplayTitle の結果）
    archived: false,
    updatedAt: 1737000000000, // Date.now()
    schema: 2
  }
  ```
  - `cloudId` は既存のローカルID（`genId()` の6文字、`ownerUid` を跨いで衝突し得る）ではなく、Firestore の `collection("trips").add()` が発行する自動IDを使う
  - ローカルの `tripsStore` エントリ（`{ id, data, archived }`）に `cloudId: string|null` と `updatedAt: number`（既定 0、`saveState()` のたびに現在のしおりのエントリへ `Date.now()` を書き込む「最終更新の目安」）を追加して対応付ける。両フィールドとも `localStorage` に永続化し、`loadStoreV2` で防御的に正規化する
  - 16 で `publicId: string|null` フィールドが追加される（公開層のドキュメントIDを端末間で引き継ぐため、この `trips/{cloudId}` ドキュメントにも保存する）
  - Firestoreセキュリティルール（設定済み想定）: `trips/{tripId}` は `read`/`update`/`delete` が `request.auth.uid == resource.data.ownerUid`、`create` が `request.auth.uid == request.resource.data.ownerUid`。`publicTrips/{publicId}` は 16 参照
  - **add の二重実行防止**: `cloudId` の採番（`add()`）が実行中のエントリに対して、採番完了を待たずに再度 `cloudUpsertEntry()` が呼ばれると `add()` が二重に走り、クラウド上にしおりが重複作成される（例: ログイン直後のマージによるアップロードが完了する前に、ユーザーが公開トグル（16）やアーカイブ操作を行った場合）。実行中の `add` を `cloudAddInFlight`（`entry.id` -> Promise）で記録し、`cloudId` が未設定かつ実行中の add がある場合は、その完了後に改めて `cloudUpsertEntry()` を呼び直す（＝2回目は `set` 経路になる）

- **同期方針**（実装はシンプルに保つ）:
  - **ログイン直後**（`onAuthStateChanged` でログイン検知した回のみ・1回限り）に `runCloudMerge()` を実行する: `where("ownerUid","==",uid)` で自分のしおりを全件取得し、ローカルの `tripsStore` とのマージ計画を純粋関数 `computeTripsMergePlan(localTrips, cloudDocs)` で組み立てる
    - `cloudId` を持つローカルエントリ: 対応するクラウド文書があれば `updatedAt` を比較し、クラウドが新しければローカルを上書き、そうでなければアップロード対象にする（対応する文書がクラウド側で見当たらない場合もアップロードして復元する）
    - `cloudId` を持たないローカルエントリ（ログイン前に作成したしおり）: 常にアップロード対象にする
    - どのローカルエントリにも対応しないクラウド文書: 新規ローカルエントリとして追加する
    - 適用後、マージで動いた件数（アップロード＋ローカル更新＋ローカル追加の合計）を「クラウドと同期しました（N件）」でトースト通知する（4言語。0件なら通知しない）
    - `computeTripsMergePlan` は Firestore呼び出しを含まない純粋関数のため、スタブ無しで単体テスト可能
  - **保存時**: `saveState()`（ローカル保存は従来どおり即時）が呼ばれるたびに、ログイン中であれば現在のしおりのエントリを**デバウンス2秒**で `cloudUpsertEntry()` により Firestore へ書き込む（`cloudId` が無ければ `add()` で新規作成して採番、あれば `set(..., {merge:true})` で更新。`updatedAt` を書き込み時刻で更新する）
  - **削除/アーカイブ**: `requestDeleteTrip` のクラウド反映は文書削除（`cloudDeleteEntry`）、`requestArchiveTrip`/`unarchiveTrip` のクラウド反映は `archived` フィールド更新（`cloudUpsertEntry` を即時呼び出し。これらの操作は必ずしも「現在のしおり」に対して行われるとは限らないため、`saveState()` 内の自動デバウンス対象とは別に明示的に呼ぶ）
  - **ログアウト時**: ローカルデータは一切削除しない。以後 `authUser` が `null` になるため、以降のクラウド書き込み・マージ処理はすべて自然に無効化される
  - **エラー処理**: 書き込み・読み込みエラー（オフライン等）は握りつぶし、連続失敗中はトーストを1回だけ表示する（成功したらフラグをリセットし、次の失敗でまた1回だけ表示する）。ローカルの動作には一切影響しない

- **UI**:
  - しおり一覧モーダルの各行のうち `cloudId` を持つものに、控えめな ☁ アイコンを表示する（`trips-list-item-cloud`）
  - ログイン中はしおり一覧モーダルの見出し下に「☁ 同期中」の控えめな表示を出す
  - 未ログインでも上記以外の全機能（項目追加・ルート検討・CSV入出力・共有・PDF出力・アーカイブ・準備モーダル・地図など）がこれまでどおり利用できる

### 16. 公開層と公開URL

14 で用意したサニタイズ関数と 15 のプライベート層の上に、**誰でも読める公開コピー**（`publicTrips` コレクション）と**短い公開URL**（`#p=<publicId>`）を実装する。これにより「本人だけが読める完全データ（`trips`）」と「誰でも読める公開コピー（`publicTrips`）」の2層構造が完成する。公開コピーには非公開マークを付けた部分が**物理的に存在しない**ため、公開側からは非公開データを取得できない。

**絶対条件**: 未ログイン時の既存の全機能がこれまでどおり動くこと。公開URL閲覧モードでローカルデータが一切汚染されないこと。

- **データ構造（Firestore）**: `publicTrips/{publicId}` ドキュメント
  ```js
  {
    ownerUid: "…",         // 書き込み権限の判定に使う（read は誰でも可）
    data: "{...}",          // JSON.stringify(sanitizeTripForPublic(tripData))。常にサニタイズ済み
    title: "東京旅行",       // tripDisplayTitle の結果
    updatedAt: 1737000000000,
    schema: 2
  }
  ```
  - `publicId` は **Firestore の自動ID**（`collection("publicTrips").add()` が発行）を使う。連番など推測可能なIDは禁止（URLを知る人だけが読める前提のため）
  - ローカルの `tripsStore` エントリに `publicId: string|null`（既定 null＝未公開）を追加する。`normalize`/`loadStoreV2`/`migrateV1` で防御的に既定 null へフォールバックし、`localStorage` に永続化する
  - 端末間で公開状態を引き継ぐため、プライベート層 `trips/{cloudId}` ドキュメントにも `publicId` を保存し、`runCloudMerge()` のマージ時にローカルへ復元する
  - Firestoreセキュリティルール（設定済み想定）: `publicTrips/{publicId}` は `read` が `true`（誰でも可）、`create`/`update` が `request.auth.uid == request.resource.data.ownerUid`、`delete` が `request.auth.uid == resource.data.ownerUid`

- **共有モーダルの再設計**（従来のハッシュ共有リンクは残し、その下に公開セクションを足す）:
  - **未ログイン時**: 従来どおりハッシュ共有リンク（`#d=`）のみ。「ログインすると短い公開リンクを発行できます」の案内を表示する（4言語）
  - **ログイン時**: 「🌐 公開する」トグルを表示する。ハッシュ共有リンク（リンクに全部埋め込む方式）と短い公開リンクの2つが併存する
    - **ON**: `publicTrips` にサニタイズ済みコピーを `add()` して `publicId` を採番・保存し、公開URL `<origin><pathname>#p=<publicId>` を生成して表示する（コピーボタン付き）。`publicId` はローカルとプライベート層の両方へ保存する
    - **OFF**: `publicTrips/{publicId}` を削除し、ローカル/クラウドの `publicId` を `null` に戻す（「公開を停止しました」トースト）
    - 公開中は「🌐 公開中」バッジと、非公開で除外された件数（`countSanitizedExclusions` を再利用）を表示する
    - 通信中はトグルを `disabled` にして二重送信を防ぐ
  - **公開中のしおりの自動更新**: 公開中のしおりを編集したら公開コピーも自動更新する。既存の cloud sync デバウンス（`saveState()` → `scheduleCloudSync()` → `cloudUpsertEntry()`）に相乗りし、書き込み成功後に `syncPublicCopyIfPublished(entry)` が `publicId` を持つエントリのみ公開コピーを上書きする。公開コピーは**常にサニタイズ済み**であること

- **公開URLの閲覧（ログイン不要・読み取り専用）**:
  - 起動時に `#p=<publicId>` があれば `publicTrips/{publicId}` を取得し、**読み取り専用モード**で表示する
  - **実装方針**: グローバルフラグ `viewOnly` を持つ。ハッシュを検出した時点で**同期的に** `viewOnly = true` にしてから非同期取得を行う（`init()` 末尾の `saveState()` を含め、以降の保存処理を確実に無効化するため）
  - **ローカル非汚染の保証（最重要）**: `saveState()` は `viewOnly` なら**冒頭で即 return** する（`localStorage`・クラウド書き込みのいずれも発生しない）。加えて各操作関数（`addItemFromForm`/`deleteItem`/`duplicateItem`/`requestDeleteDay`/`onDragHandlePointerDown`/`handleGmapChange`/`runRouteCalculation`/`runMapUpdate`/`applyTextImport`/`addChecklistItem`/`deleteChecklistItem`/`switchTrip`/`createNewTrip`/`requestDeleteTrip`/`requestArchiveTrip`/`unarchiveTrip`/`openTripsModal`/`openShareModal`/`publishCurrentTrip`/`unpublishCurrentTrip`）の冒頭にも `viewOnly` ガードを置く（多層防御）
  - `trip` 変数は一時的に他人の公開コピーを指すが、`tripsStore` / `currentTripId` は自分のローカルデータのまま変更しない。`runCloudMerge` のローカル更新も `viewOnly` 中は `trip` を差し替えない
  - **編集系UIの無効化/非表示**:
    - 入力欄は `readOnly`、セレクト・チェックボックス・追加ボタンは `disabled` にする（`applyViewOnlyUI()` と各ビルド関数）。タイトルは `setAttribute("contenteditable", "false")`（属性/プロパティ反映の環境差を避けるためプロパティ代入は使わない）
    - ボタン類（追加フォーム・チェックリスト追加欄・ルート検討・地図更新・共有・CSV入出力・しおり一覧・準備・設定・ログイン・日追加/削除・ドラッグハンドル・複製・削除・🔒トグル）は `body.view-only-mode` スコープのCSSで一括非表示にする
  - ヘッダーに「👁 閲覧モード」バッジと「自分のしおりに戻る」ボタンを表示する（4言語）。戻るとハッシュを消して `trip` をローカルの現在のしおりへ戻し、通常モードに復帰する
  - 言語切替・地図表示・PDF出力・タブ切替（日の閲覧）は使えること。なお閲覧中の言語切替は一時オブジェクトの `trip.lang` を変えるだけで永続化されないため、「自分のしおりに戻る」と自分の保存言語へ戻る（ローカル非汚染を優先した意図的な挙動）
  - **取得失敗時**（ドキュメントが存在しない・削除済み・データ破損・オフライン・権限エラー・Firebase 未初期化）: `viewOnly` を解除し、ハッシュを消して「このしおりは公開されていません」トースト（4言語）を出し、**通常モードで起動する**。この場合 `init()` 末尾の `saveState()` はスキップ済みのため、起動時の移行・正規化結果の永続化をフォールバック側で肩代わりする
  - **XSS**: 公開データは他人が作った可能性がある前提で扱う。必ず `normalizeTrip` を通し、描画は `textContent` / `createElement` のみ（`innerHTML` への連結禁止）。`gmap` の危険スキームは既存の `isSafeHttpUrl` で弾く

- **しおり一覧**: `publicId` を持つ行に 🌐 バッジを表示する（`trip-list-item-public`）

- **エラー処理**: Firestore の読み書きは失敗しうる（オフライン・権限）。失敗しても通常モードで起動し、アプリはローカル動作のみで完全に機能する（15 の `handleCloudError` を共用し、連続失敗中のトーストは1回だけ）

### 17. 開始時刻の手動固定

各項目の時刻は基本的に「その日の開始時刻 + 各項目の所要時間の累積」で全自動計算される（3参照）。しかし「このフライトは08:55発」「チェックインは15:00」のような動かせない時刻を表現できるよう、項目ごとに開始時刻を手動で固定できるようにする。固定した項目は新たな起点になり、以降の項目はそこから再計算される。

- **データ**: 行程項目（move含む全カテゴリー）に `fixedStart: string|null` を追加する（`"HH:MM"` 形式。既定 `null`＝未固定）
  - `normalizeTrip` で防御的に正規化する（`normalizeFixedStart` ヘルパー）。`/^\d{1,2}:\d{2}$/` にマッチしない文字列・文字列以外の型は `null` にフォールバックする。マッチした場合も時=0-23・分=0-59の範囲外は `null` にする。有効な値はゼロ埋め2桁の正規形（例 `"9:5"` → `"09:05"`）に変換する
  - 新規作成される項目（追加フォーム・サンプルしおり・ルート検討ボタンが生成する自動 move）は既定 `fixedStart: null` で作成される。複製ボタンは深いコピーのため `fixedStart` もそのまま複製される
  - サンプルしおりの「三井ガーデンホテル上野」（チェックイン15:00という既存のメモ文言を持つ項目）は、本機能のデモとして例外的に `fixedStart: "15:00"` を初期値に持つ

- **時刻計算（`getDayTimedItems` の拡張 — 中核）**:
  - `item.fixedStart` が設定されている項目は、その項目の `startMin` を cursor（それまでの累積）ではなく固定時刻から決める: `startMin = Math.floor(cursorBefore / 1440) * 1440 + parseTimeToMinutes(item.fixedStart)`（`cursorBefore` はその項目に来る前の cursor の値）。つまり固定時刻は「cursor が現在いる暦日と同じ日」の時刻として解釈される（日跨ぎで cursor が1440を超えていれば、固定時刻もその翌日側に乗る）
  - `endMin = startMin + dur` とし、`cursor = endMin` として以降の項目へ継続する（＝固定項目が新たな起点になる）
  - **時差との整合**: cursor は時差対応（13）によりオフセット加算後の値であるため、固定時刻はそのまま「その時点で画面に表示されている現地時間」として解釈される（上記の式の上で自然に成立する。固定時刻自体に時差を二重適用することはない）
  - **競合（重なり）の検出**: 固定により `startMin < cursorBefore`（＝直前の項目の終了時刻より前）になる場合、その項目に `conflict: true` を立てて返す。データは削除・調整しない（UIで警告表示のみ）
  - `getDayTimedItems` の戻り値は `day.items` と常に1:1のインデックス対応を保つ（行程番号バッジ・地図ピンの番号・PDF出力ビューがこの対応関係に依存しているため、要素を増減させたり順序を変えたりしない。13と同じ制約）
  - 既存の「1440分超で `(+n)` を付けて表示する」ロジックはそのまま維持する
  - `fixedStart` が未設定のときは従来と完全に同じ計算結果になる（既存しおりへの影響ゼロ）

- **UI（タイムラインカード）**:
  - 時刻列の開始時刻部分をクリック可能な `<button>` にする（`.item-start-time-btn`）。クリックすると同じ位置に `<input type="time">` がインライン表示され、現在の計算値（`startMin` を暦日境界内の `HH:MM` に変換した値。オーバーフロー表記 `(+n)` は含めない）がプリセットされる
  - 値を確定（`change`）すると `item.fixedStart` に正規化済みの値を保存し、以降の時刻が再計算される。値を変えずにフォーカスを外す（`blur`）か `Escape` キーで編集をキャンセルする（未確定のまま閉じる）
  - 固定中は開始時刻に 📌 アイコンを付けて強調色（`--pink-dark`）で表示し、隣に固定解除用の小さな ✕ ボタン（`.item-fixed-clear-btn`）を表示する。押すと `item.fixedStart = null` に戻し、自動計算へ復帰する
  - 未固定の開始時刻は従来どおりの見た目だが、ホバーで背景がわずかに変化しクリックできることを示す
  - **重なり警告**: `conflict: true` の項目には「⚠ 前の予定と重なっています」を常時表示する（4言語。既存の `.item-unresolved-msg` を参考にした控えめな警告色 `--warn`）
  - viewOnly（公開URL閲覧、16参照）モードでは編集不可にする: 開始時刻は `<button>` ではなく `<span>`（クリックハンドラなし）として描画し、固定解除ボタンは描画しない。既存の `.view-only-mode` ガード方針（各操作関数の冒頭 `if (viewOnly) return;` によるJS側ガード）に従う

- **印刷ビュー（12）への適用**: `fixedStart` が設定された項目は、時刻表示の先頭に 📌 を付ける（`buildPrintItemRow`）。時刻自体は `getDayTimedItems` が計算済みの値をそのまま使う

- **CSV（8）への適用**:
  - 行程テーブルのヘッダーに任意列 `fixedStart`（`"HH:MM"` または空文字）を追加する（`tz`/`arriveTz`/`private` 等と同様、必須列ではなく各項目行の値を持つ）
  - **後方互換必須**: この列を持たない旧形式のCSVを読み込んでもエラーにならず、`fixedStart` は列が無い場合 `null` として扱う（列の有無は `colIndex.fixedStart != null` で判定し、必須列の判定ロジックには含めない）。列の値も `normalizeFixedStart` を通すため、`"HH:MM"` 形式以外の壊れた値が入っていても防御的に `null` になる
  - CSVエクスポートは自分用の完全バックアップのため、`fixedStart` もそのまま出力する

- **共有リンク・公開URL（7・16）への適用**: `fixedStart` は非公開情報ではないため `sanitizeTripForPublic` での除去対象にしない（他のフラグと異なり削除しない）。共有リンク・公開コピーのいずれにもそのまま含まれ、閲覧側でも固定時刻どおりに表示される

- **ルート検討ボタン（6）との整合**: ルート検討の再計算は自動生成 move（`auto: true`）のみを削除・再挿入する既存仕様のため、滞在地側の `fixedStart` が失われることはない

### 18. 編集できる共有リンク

15（プライベート層）・16（公開層）に続く3つ目の層として、**ログイン不要で誰でも編集できる共有リンク**（`editTrips` コレクション、`#e=<editId>`）を実装する。オーナー（しおりの持ち主）は自分の完全データを保ったまま、非公開項目を含まない編集用コピーを共同編集者と双方向にリアルタイム同期する。

**絶対条件**: 未ログイン・通常利用の既存機能が壊れないこと。共同編集モード（`collabMode`）中に共同編集者のローカルデータ（`localStorage`・自分の `trips`）が一切汚染されないこと。マージ処理で非公開項目・非公開の日・非公開メモ・非公開リストを絶対に失わないこと。

- **データ構造（Firestore）**: `editTrips/{editId}` ドキュメント
  ```js
  {
    ownerUid: "…",          // 変更不可（Firestoreルールで固定）。read/updateは誰でも可、create/deleteはオーナーのみ
    data: "{...}",           // JSON.stringify(sanitizeTripForPublic(tripData))。常にサニタイズ済み（200,000バイト未満）
    title: "東京旅行",
    updatedAt: 1737000000000,
    writerId: "…",           // 直近の書き込みを行ったセッションのランダムID（自分自身のエコー無視に使う）
    schema: 2
  }
  ```
  - `editId` は **Firestore の自動ID**（`collection("editTrips").add()` が発行）を使う。これがそのまま「合言葉」になる
  - Firestoreセキュリティルール（設定済み・変更不可）: `get` は誰でも可、`list` は不可、`create`/`delete` はオーナーのみ（`request.auth.uid == ownerUid`）、`update` は**未認証でも可**（URLを知っていれば編集可能にするため）だが `ownerUid` は変更不可・`data` は200,000バイト未満の文字列という制約を持つ
  - ローカルの `tripsStore` エントリに `editId: string|null`（既定 null）を追加する。`normalizeTrip`/`loadStoreV2`/`migrateV1` で防御的に既定 null へフォールバックし、プライベート層 `trips/{cloudId}` にも保存して端末間で引き継ぐ
  - 各セッション（タブ）は起動時にランダムな `SESSION_WRITER_ID` を1つ持ち、`editTrips` への全ての書き込みにこれを `writerId` として埋め込む。`onSnapshot` で自分自身が書いた変更のエコーを受信したときはこれで判別して無視する（無限ループ防止）
  - `days`/行程項目/持ち物/やること各要素は安定した `id` を持つ（`day.id` も本機能のためにオーナー側で発行・保持する。旧データには無いため `normalizeTrip` が防御的に発行する）。マージのアンカーに使うため、これらの `id` は `sanitizeTripForPublic` で除去しない

- **共有モーダルの拡張**（16の「🌐 公開する」セクションの下に追加。両方同時ONも可、独立して動作する）:
  - **未ログイン時**: トグルを表示せず「ログインすると編集できるリンクを発行できます」の案内のみ表示する（4言語）
  - **ログイン時**: 「✏️ 編集できるリンクを発行」トグルを表示する
    - トグルの直下に**警告文を常時表示**する（4言語）: 「⚠️ このリンクを知っている人は誰でも編集できます。」「🔒非公開の項目は含まれません。」
    - **ON**: `editTrips` にサニタイズ済みコピーを `add()` して `editId` を採番・保存し、編集URL `<origin><pathname>#e=<editId>` を生成して表示する（コピーボタン付き）。`editId` はローカルとプライベート層の両方へ保存する。サニタイズ後データが200,000バイト以上ならトーストで通知し発行しない
    - **OFF**: `editTrips/{editId}` を削除し、ローカル/クラウドの `editId` を `null` に戻す（「編集できるリンクを停止しました」トースト）
    - 発行中は「✏️ 発行中」バッジを表示する。通信中はトグルを `disabled` にして二重送信を防ぐ

- **オーナー側の双方向同期**:
  - **push（オーナー→editTrips）**: 16 の `syncPublicCopyIfPublished` と同じ経路（`cloudUpsertEntry` 成功後）に相乗りする `syncEditCopyIfEnabled(entry)` が、`editId` を持つエントリのみサニタイズ済みコピーを `set(..., {merge:true})` する。`writerId` に自セッションIDを入れる。データが200,000バイト以上ならトーストで通知し書き込みをスキップする（連続発生時は1回だけ）
  - **pull（editTrips→オーナー）**: 現在開いているしおりが `editId` を持つ間だけ `editTrips/{editId}` を `onSnapshot` 購読する（`syncEditListenerForCurrentTrip`）。しおり切替・アーカイブでの切替・編集リンクの発行/停止のたびに購読を張り直し、以前の購読は必ず解除する
    - 受信した `writerId` が自分の `SESSION_WRITER_ID` と一致する場合は自分自身の書き込みのエコーなので無視する
    - それ以外は下記のマージを適用し、`persistLocalOnly()` + `render()` で反映する。**マージ適用自体は `saveState()` を呼ばず、`editTrips` への push を誘発しない**（ループ防止。専用フラグ `applyingRemoteEditUpdate` で処理区間を明示する）
    - `viewOnly`/`collabMode` 中はオーナー側リスナーを持たない（自分のしおりを見ていない、または他人のしおりを編集中のため）

- **マージ（オーナー側の受信データ反映。中核）**: `mergeRemoteEditIntoOwnerTrip(ownerTrip, receivedData)` として **Firestore呼び出しを含まない純粋関数**に切り出す（自動テストで直接検証できるよう `window.__tabiShioriCollabInternals` に公開）。
  - **アンカー方式**: 非公開要素の位置情報を別途永続化する必要はない。オーナーの現在の完全データ（`ownerTrip`）を今この瞬間に `sanitizeTripForPublic` した結果（`ownerPublic`）と比較するだけで、「どの要素が非公開で共同編集者から見えていないか」と「その要素の直前にあった、共同編集者にも見えていたはずの要素（アンカー）」を毎回再計算できる。アンカーは公開データには一切含まれず、オーナー側のこの計算にしか存在しない
  - 手順:
    1. 受信データ（`normalizeTrip` 済み）の `days`/`items`/`packing`/`todos` を正として採用する
    2. `ownerPublic` に含まれない `ownerTrip` の日（`day.priv:true` で除外された日）を、アンカー（直前の可視な日の `id`。先頭なら `null`）の直後に挿し戻す。アンカーが受信データ側に見当たらない（＝共同編集者がそのアンカーごと削除した）場合は末尾に追加する。日の同定には安定な `day.id` を使う
    3. 各日について、`ownerPublic` に含まれない項目（`priv:true` の項目、および非公開項目に隣接して除外された自動生成move）を、同様にアンカー（直前の可視な項目の `id`）の直後（無ければ末尾）に挿し戻す
    4. `notePriv:true` だった項目は、受信データの空メモで上書きせず、**オーナーの元メモ**を保持する（`notePriv` フラグ自体も復元する）
    5. `packingPriv`/`todosPriv` が `true` のリストは、受信データ（常に空のはず）で上書きせず**オーナーの元リストをそのまま保つ**
    6. `packingPriv`/`todosPriv` が `false` のリストは、要素単位の `priv:true` を項目と同じアンカー方式で挿し戻す
    7. `title`/`titles`/`lang`、可視な日・項目のその他のフィールド（`date`/`startTime`/`tz`/`dateManual`/`name`/`note`/`fixedStart`/`gmap` 等）は受信データ（共同編集者の最新の編集）をそのまま採用する
    8. 最終結果は必ず `normalizeTrip` を通してから採用する（受信データは他人が編集した可能性がある前提のため）
  - オーナー自身の `priv`/`notePriv`/`packingPriv`/`todosPriv`/`dateManual` 等、公開データに含まれないフラグは、上記の手順により失われない

- **共同編集者側（`collabMode`）: `#e=<editId>` で開いたとき**:
  - `viewOnly` とは独立した新モード（`collabMode`）。ハッシュを検出した時点で**同期的に** `collabMode = true` にしてから `editTrips/{editId}` を非同期取得する（`checkPublicHash` と同じ実装方針）
  - 取得成功: `normalizeTrip` を通して `trip` に採用し表示する。**`localStorage` には一切保存しない**（`tripsStore`/`currentTripId` は自分のローカルデータのまま変更しない。相手のしおりでローカルを汚染しないための最重要ガード）
  - **保存経路の分岐（実装の要）**: ほぼ全ての編集操作の末尾が呼ぶ `saveState()` に `collabMode` の分岐を1つ追加するだけで実現する。`viewOnly` なら従来どおり no-op、`collabMode` ならローカル保存・自分のクラウド同期を一切せず、代わりに `scheduleCollabPush()`（デバウンス2秒）で `editTrips/{editId}` に `set(..., {merge:true})` する。`writerId` に自セッションID、`ownerUid` は受信時に控えた値をそのまま送り返す（ルール制約でオーナーは変更不可のため）。200,000バイト以上なら push をスキップしトースト通知する
  - **pull**: `onSnapshot` で他の共同編集者・オーナーからの更新を受信する。自分の `writerId` のエコーは無視する。共同編集者側のローカルコピーはもともとサニタイズ済みデータそのもの（非公開要素を一切持たない）なので、オーナー側のような非公開保護マージは不要で、受信データへの**全置換**でよい（`normalizeTrip` を通す）
  - **ログイン/クラウド同期系は `collabMode` で無効化**する（`runCloudMerge`/`cloudUpsertEntry`/`scheduleCloudSync` を `collabMode` 中はガードし、自分の `trips` には一切書き込まない）。ログイン自体は可能（ログインしても何も汚染しない）。「自分のしおりに戻る」で通常モードに復帰した際、ログイン中なら `runCloudMerge()` を再開する
  - ヘッダーに「✏️ 共同編集中」バッジと「自分のしおりに戻る」ボタンを表示する（4言語）。戻るとハッシュを消して購読を解除し、`trip` をローカルの現在のしおりへ戻す
  - しおり一覧・共有・非公開🔒トグル各種（日単位・項目単位・リスト単位）は、自分のローカルしおりに作用してしまうため非表示にする（`body.collab-mode` スコープのCSS。JS側の関数ガードと合わせた多層防御）
  - **取得失敗時**（ドキュメントが存在しない・削除済み・データ破損・オフライン・Firebase 未初期化）: `collabMode` を解除し、ハッシュを消して「このリンクは無効です」トースト（4言語）を出し、通常モードで起動する
  - **オーナーがリンクを停止した場合**: `onSnapshot` がドキュメント削除を検知し、「オーナーがこの編集リンクを停止しました」トーストを出す。以降の push は行わない（`collabRevoked` フラグ）
  - **XSS**: 受信データは他人（オーナーや他の共同編集者）が作った可能性がある前提のため、必ず `normalizeTrip` を通し、描画は既存どおり `textContent`/`createElement` のみ

- **しおり一覧**: `editId` を持つ行に ✏️ バッジを表示する（`trip-list-item-edit`）

- **競合**: 複数人が同時に編集した場合は「最後の書き込み優先」とする（デバウンス2秒のため、ほぼ同時の編集は後勝ち）。厳密な競合解決（OT/CRDT等）は行わない

- **エラー処理**: Firestore の読み書きは失敗しうる（オフライン・権限・データサイズ超過）。失敗しても通常モードで起動し、アプリはローカル動作のみで完全に機能する（15 の `handleCloudError` を共用し、連続失敗中のトーストは1回だけ）

- 言語切替は即時反映され、trip.lang に保存される
- `<html lang>` も切替に追随

## 初期データ

初回起動時（localStorage が空のとき）は、使い方がわかるサンプルしおり（Day 1 に観光2件+食事1件+宿泊1件、うち1区間に move 入り）を日本語で投入する。
場所名はすべてジオコーディングで一意に見つかる実在の都内スポットにする（宿泊は「三井ガーデンホテル上野」。「上野ホテル」のような曖昧な名前は別地域の同名施設にヒットするため使わない）。

## 実装上の注意

- すべての confirm/alert は window標準ではなく、アプリ内のモーダル/トーストで統一（翻訳できるように）
- id 生成は `Math.random().toString(36).slice(2, 8)`
- 描画は「state 変更 → render() で選択日のタイムラインを全再描画」の単純方式で良い。ただし input 編集中に再描画でフォーカスを失わないよう、blur/change 時に state 反映→再描画とする
- XSS対策: ユーザー入力は必ず textContent / createElement で挿入（innerHTML に連結しない）
- エラー時（Nominatim 到達不可など）はトーストで通知し、アプリは壊れないこと
